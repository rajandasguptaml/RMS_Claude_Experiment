1. EXECUTIVE SUMMARY
The Room Registration module is the core transactional screen within the Cubix HMS Front
Office suite. It enables front desk agents to formally check-in both walk-in and pre-booked
(reservation-linked) guests by creating a Room Registration record identified by a unique
system-generated number in the format RR########.
The module is structured as a 6-element tab wizard: Registration, Guest Details,
Complimentary Item, Others Information, Search, and Blank Registration Card. The workflow
captures room assignment, occupancy configuration, rate calculation, complete guest
demographics, identity documentation, complimentary service selections, departure
logistics, credit card guarantee details, and supplemental classification data — all within a
single, integrated interface.
Upon completion, the front desk agent activates the global Check-in button, which
transitions the guest status to "In-House", updates room inventory, opens the guest billing
account, and posts initial room charges. The Search sub-module provides multi-field search
across all historical and active registration records with per-record actions including Edit, Re-
activate, Bill Preview, Split Bill Preview, and Registration Card printing.
Key confirmed business rules include: enforced adult-to-guest-record count matching
before check-in; manager/supervisor approval with room availability revalidation for re-
activation; PCI-DSS compliant card data handling with tokenization; package-driven
mandatory complimentary item defaults; optimistic-lock concurrency control for room
assignment; a new digital marketing channel-discovery field; and conditional mandatory
logic for Meal Plan and Reference depending on booking type (reservation-linked vs. walk-in).

3. FUNCTIONAL REQUIREMENTS
FR-001 — Tab Navigation and Wizard Flow
Unique ID: FR-001 | Actor: Front Desk Agent
Description: The Room Registration screen is organized as a 6-element tab strip providing a
guided data-entry workflow. Navigation between tabs is free and non-sequential.
Preconditions: User is authenticated and has navigated to Front Office → Room Registration.
Trigger: User lands on the Room Registration page URL.
Step-by-step Flow:
1. System renders the page with the Registration tab active by default.
2. Tab strip displays six elements: Registration | Guest Details | Complimentary Item |
Others Information | Search | Blank Registration Card.
3. The first four are content tabs; Search is a display-and-action tab; Blank Registration
Card is a direct-action link opening a new browser tab.
4. User may click any tab at any time — no sequential enforcement at the navigation
level.
5. A global Check-in button (id="check-in") is fixed below all tab content and remains
visible at all times.
6. Clicking Blank Registration Card opens /front_office/room-registration/blank-
registration-card in a new browser tab.
System Response: Active tab content is shown; all other panes are hidden via CSS. Check-in
button always accessible.
Alternate/Exception Flow: Clicking Check-in before completing required fields triggers cross-
tab field validation, surfacing errors on specific fields regardless of active tab.
FR-002 — Registration Tab: Header Section
Unique ID: FR-002 | Actor: Front Desk Agent
Description: Defines booking context: optional reservation linkage, stay dates/times,
company billing association, and currency settings.
Trigger: Agent begins a new room registration (walk-in or reservation-linked).
Step-by-step Flow:
1. Reservation (optional): Agent checks the Reservation checkbox to enable the
reservation dropdown and selects a pending reservation number. System auto-populates
Check-In Date, Departure Date, and Total Nights. For walk-in registrations the checkbox
remains unchecked.
2. Check-In Date (required): Defaults to application date. Agent may change via date
picker.
=== PAGE 4 ===
3. Check-In Time: Defaults to 02:00 PM. Agent may change via time picker.
4. Departure Date (required): Defaults to next calendar day. Must be strictly after
Check-In Date.
5. Checkout Time: Defaults to 12:00 PM.
6. Total Nights: Auto-calculated (Departure Date minus Check-In Date). Checkbox
allows manual override.
7. Listed Company (optional): Checking enables the company dropdown and reveals
four conditional fields (see FR-003).
8. Currency Name (required): Defaults to BDT. Selectable from currency master.
9. Conversion Rate: Read-only; auto-populated from currency master. Must be non-
zero.
System Response: Total Nights recalculates dynamically. Currency rate fetched on currency
selection.
Alternate/Exception Flow: Check-In Date ≥ Departure Date → validation error. No
reservation selected → walk-in path; Meal Plan and Reference become optional (BR-020).
Conversion Rate zero → system prompts to verify currency master.

FR-013 — Check-in Action (Global)
Unique ID: FR-013 | Actor: Front Desk Agent
Description: The globally visible Check-in button finalizes the Room Registration, creates the
guest billing account, and transitions status to In-House.
Preconditions: At least one room added; guest record count = total adult count; all
conditionally required fields complete per booking type.
Trigger: Agent clicks the global Check-in button (id="check-in", green, below all tab panes).
Step-by-step Flow:
1. System executes cross-tab mandatory field validation.
2. System validates guest record count equals total declared adult count.
3. For reservation-linked registrations: validates Meal Plan and Reference are selected.
=== PAGE 10 ===
4. System performs optimistic concurrency lock check on the room.
5. System invokes PCI-DSS tokenization service for card data (if provided).
6. System generates Registration Number: RR + 8-digit zero-padded sequential integer
(e.g., RR00002293). Immutable after generation.
7. System persists full registration record with status "In-House."
8. System updates Room Inventory: room(s) status set to "Occupied."
9. System opens guest billing account and posts initial room charge.
10. System writes audit log: User ID, Username, timestamp, action = "Check-in",
Registration ID.
11. System releases concurrency lock on the room.
12. System displays success confirmation with generated Registration Number.

FR-015 — Re-activate Registration
Unique ID: FR-015 | Actor: Manager / Supervisor (Front Desk Agents do NOT have this
permission)
Description: Reinstates a Checked-Out registration to In-House status, subject to role-based
approval, room availability revalidation, and date validation.
Preconditions: Registration status is "Checked Out." Acting user has Manager or Supervisor
role.
Step-by-step Flow:
1. System presents re-activation confirmation dialog: Registration No., Guest Name,
Room Info, original Check-In/Check-Out dates.
2. System prompts for manager/supervisor credentials or approval code.
3. System validates approving user has Manager or Supervisor role.
4. System revalidates room availability: room must not be assigned to another active
In-House registration for any overlapping date range.
5. System validates stay dates fall within the operationally permissible window
(configurable threshold).
6. If all checks pass: status set to "In-House"; Room Inventory updated to "Occupied."
7. Audit log written: Approving User ID, Username, timestamp, action = "Re-activate",
Registration ID.
8. Success confirmation displayed.

6. BUSINESS RULES
ID Rule Detail
BR-
001
Mandatory Field
Enforcement
All fields marked * are mandatory. System must prevent Check-in
and highlight offending fields across all tabs if any required field is
empty.
BR-
002
Date Sequence
Validation
Check-In Date must be strictly less than Departure Date. If equal or
reversed, system displays a validation error and blocks registration.
Total Nights auto-calculates as Departure Date minus Check-In Date
in calendar days.
BR-
003
Reservation Linkage
(Conditional)
The Reservation dropdown is only active when the Reservation
checkbox is checked. Selecting a reservation auto-populates Check-
In Date, Departure Date, and Total Nights. Auto-populated values
may be overridden by the agent.
BR-
004
Listed Company Fields
(Conditional Display)
Contact Person, Mobile Number, Payment Mode, and Pay For fields
are hidden (display:none) by default. They become visible and
editable only when the Listed Company checkbox is checked and a
company is selected (CSS class: listed_company_required_fields).
BR-
005
Room Number
Assignment
Room Number can only be selected via the Room List modal; it
cannot be typed manually. The modal filters rooms by selected
Room Type and current date range availability. Rooms with active
In-House registrations for overlapping dates are excluded.
BR-
006
Rate Calculation Logic Total Room Rent Amount = Rack Rate − Discount + Service Charge +
VAT Amount + City Charge + Additional Charges. Fixed discount is
subtracted as an absolute value; Percent discount is calculated as a
percentage of Rack Rate. Discount Amount must not exceed Rack
Rate.
BR-
007
RRC Reverse Calculation When a target Total Room Rent Amount is entered in the RRC
modal, the system reverse-calculates all selected charge
components. Each component has an inclusion checkbox. Clicking
Apply Rate pushes the back-calculated values to the main form
fields.
BR-
008
Per-Room Dates vs.
Global Dates
When "Same as global date" is checked (default), per-room Check-
In and Departure dates are locked to and auto-sync with header
dates. When unchecked, per-room dates are independently
editable.
BR-
009
Room Status Flags No Post suppresses automated nightly charge posting.
Complimentary suppresses all room charges. House Use marks the
room as internal hotel use. These flags are independent checkboxes
but have mutually exclusive billing implications; business process
must define conflict resolution if multiple are checked.
BR-
010
Guest Count
Enforcement
(CONFIRMED)
The total number of saved Guest Detail records must equal the sum
of Adult (Per Room) counts across all room rows in the registration.
This is enforced at Check-in. If counts do not match, Check-in is
blocked with the message: "Guest records must match the declared
adult count. Please complete Guest Details."
BR- Country and Nationality Country defaults to "Bangladesh." Nationality defaults to
=== PAGE 26 ===
011 Defaults "Bangladeshi." Both are editable by the agent.
BR-
012
Returning Guest Search The Search button on Guest Details queries the guest profile store
by name and/or date of birth. A successful match pre-fills all form
fields from the stored profile, avoiding duplicate data entry. Agent
may override any pre-filled values.
BR-
013
Blocked Guest Warning If a guest profile retrieved via the Search function has the Blocked
Guest flag set, the system displays a prominent warning to the
agent before any further action. The agent must explicitly
acknowledge the warning to proceed. A management override may
be required (to be confirmed in process documentation).
BR-
014
Re-activate Registration
(CONFIRMED)
Re-activation requires: (1) Manager or Supervisor role
credentials/approval code; (2) room availability revalidation (no
overlapping In-House assignment); (3) date validity check within a
configurable threshold. Front Desk Agents cannot approve re-
activation. All checks must pass before status is restored to "In-
House." The approving user's identity is captured in the audit log.
BR-
015
Complimentary Item
Defaults and Mandatory
Items (CONFIRMED)
Complimentary items are pre-checked based on the selected Meal
Plan/Room Type package configuration. Items designated as
mandatory in the package definition are rendered as HTML disabled
checkboxes and cannot be deselected by the agent. A registration
may have zero complimentary items if the selected package has no
mandatory items and the agent deselects all optional items.
BR-
016
Credit Card Optional
(CONFIRMED)
Credit card capture is not mandatory. Registrations may proceed
without card data for cash-paying or company-billed guests. The
Credit Card Information section contains no required (*) fields.
BR-
017
Registration Number
Auto-Generation
On successful Check-in, the system auto-generates a Registration
Number in the format RR + 8-digit zero-padded sequential integer
(e.g., RR00002293). This number is immutable after creation and
serves as the primary display identifier.
BR-
018
Check-in Button Global
Scope
The Check-in button (id="check-in") is rendered outside all tab
panes and is always visible. It executes cross-tab validation across
all tabs regardless of which tab is currently active.
BR-
019
PCI-DSS Credit Card
Compliance
(CONFIRMED)
Raw credit card numbers are never stored in any database table, log
file, or audit record. Card number input is masked on-screen
(displaying last 4 digits only after entry). Raw card data is
transmitted only to the PCI-DSS tokenization service over an
encrypted TLS channel. The tokenization service returns a token,
which is stored in the card_token field. Only card_last_four (4
digits), card_type, card_holder_name, card_expiry_date, and
card_reference are stored in plain form.
BR-
020
Conditional Mandatory
Fields for Booking Type
(CONFIRMED)
Meal Plan and Reference are mandatory ONLY for reservation-
linked registrations (when a Reservation is selected). For walk-in
registrations (no reservation linked), both fields are optional. The
system must apply this conditional validation dynamically based on
the registration type.
BR-
021
Room Assignment
Concurrency Control
(CONFIRMED)
When an agent selects a room from the Room List modal, the
system acquires an optimistic concurrency lock (UUID token stored
in concurrency_token field). If two agents attempt to check in the
same room concurrently, only the first completed Check-in
=== PAGE 27 ===
transaction succeeds. The second agent receives: "Room [X] is no
longer available. Please select a different room." The lock is
released after successful commit or if the session is abandoned.
BR-
022
Digital Channel
Discovery Tracking
(CONFIRMED/NEW)
The field "How did you find out about our hotel?" is added as a
digital checkbox group in the Others Information tab (options:
Facebook / Website / Google / Others). This data is stored in the
channel_discovery SET field and is available for marketing analytics
reports and campaign attribution dashboards. The same field
appears on the Blank Registration Card as a print checkbox for
manual capture.
BR-
023
Currency Defaulting Currency defaults to BDT with Conversion Rate 1.00. If a foreign
currency is selected, the Conversion Rate must be non-zero. Walk-in
guests using foreign currency must have a valid conversion rate
configured in the currency master before registration can proceed.
BR-
024
Audit Trail All registration events (create, edit, check-in, re-activation, check-
out) must be logged with: User ID, Username, timestamp, action
type, and Registration ID. The CheckIn By and CheckOut By
usernames are displayed in the Search grid status column for
operational audit visibility.
7. NON-FUNCTIONAL REQUIREMENTS
ID Category Requirement
NFR-
001
Performance The Room Registration page must load within 3 seconds on a standard
broadband connection. The Room List modal must populate available
rooms within 2 seconds of selecting a Room Type. The Search tab
must return results within 3 seconds for any filter combination across
the full 2,291+ record dataset. DataTable client-side filtering must
respond in under 500ms. The Check-in action must complete within 5
seconds end-to-end including tokenization service call.
NFR-
002
Security — PCI-DSS All credit card data must comply with PCI-DSS standards. Card
numbers are tokenized via an approved tokenization service before
persistence. Only the token, last 4 digits, card type, holder name,
expiry, and reference are stored. No raw card number is written to
databases, logs, or audit trails. The tokenization service connection
must use TLS 1.2 or higher.
NFR-
003
Security — CSRF All form submissions must include a valid CSRF token (csrf_test_name
hidden field observed in DOM). The server must validate the CSRF
token on every POST request.
NFR-
004
Security — RBAC Access to Room Registration is restricted to authenticated users with
the Front Office role. The Re-activate Registration action is restricted
to Manager and Supervisor roles. Front Desk Agents must not be able
to trigger re-activation. Role enforcement must be implemented
server-side, not only in the UI.
NFR-
005
Security — Data
Privacy
Sensitive guest data (passport number, visa number, National ID,
driving license) must be masked or encrypted at rest. Guest document
files must be stored in a secure, access-controlled file storage system.
Guest document paths must not be publicly accessible via direct URL
=== PAGE 28 ===
without authentication.
NFR-
006
Security — File
Upload
The Guest Document upload must validate: file type (JPEG, PNG, PDF
only), maximum file size (to be defined by system configuration), and
must scan for malware before storage. Uploaded files must be
renamed to a server-generated UUID to prevent directory traversal
attacks.
NFR-
007
Usability Required fields must be clearly marked with an asterisk (*). Validation
errors must be displayed inline adjacent to the offending field. Default
values (Check-In Date = application date, Total Nights = 1, Country =
Bangladesh, Currency = BDT, Adult = 1, Child = 0, Discount Amount =
0) must reduce manual entry effort. The tab navigation must not
enforce sequential completion. The Check-in button must use a
visually distinct color (green) and be consistently visible across all tabs.
NFR-
008
Usability — Date
Format
Date input fields must use a consistent format system-wide. A
discrepancy exists between YYYY-MM-DD in form inputs and
DD/MM/YYYY on the Blank Registration Card. Standardize to the
hotel's configured locale format across all screens and documents.
NFR-
009
Scalability The system must support concurrent registration sessions from
multiple front desk terminals without data conflicts. Room availability
checks and concurrency locks must be atomic to prevent race
conditions. The DataTable in the Search tab must handle 10,000+
records without degrading below the 3-second response target.
NFR-
010
Concurrency Room assignment concurrency control must be implemented via
optimistic locking (UUID token per room-registration attempt). The
lock must be released automatically if the session is abandoned (e.g.,
browser closed, session timeout) within a configurable timeout
window (e.g., 10 minutes).
NFR-
011
Audit Trail All registration events (create, edit, check-in, re-activate, check-out)
must be logged immutably with User ID, Username, timestamp, action
type, and registration ID. Logs must be retained for a minimum period
as defined by hotel policy and applicable regulations.
NFR-
012
Print/Export The Blank Registration Card and per-guest Registration Card must
render correctly in browser print mode (Ctrl+P) with appropriate
margins and page breaks. Bill Preview and Registration Card links must
render within 2 seconds. CSS must include print-optimized styles
(@media print).
NFR-
013
Browser
Compatibility
The module must function correctly on the latest versions of Chrome,
Firefox, Edge, and Safari. Time pickers and date pickers must work
cross-browser without polyfills.
NFR-
014
Internationalization The Currency master supports multiple currencies with conversion
rates. The Airline master contains 65 international airlines. The
Country master supports international country selection. Date formats
and number formats should respect the hotel's locale configuration.

9. ACCEPTANCE CRITERIA
All criteria use Given/When/Then format. Each criterion must pass for the module to be
considered ready for production.
AC-001 — New Walk-in Registration (End-to-End)
Given a front desk agent is authenticated and on the Room Registration page with no
reservation linked
When the agent fills Check-In Date, Departure Date, selects a Room Type, selects a Room
Number via Room List, enters Adult count = 1, selects Meal Plan (optional for walk-in), adds
at least 1 Guest Detail record (Title + First Name + Full Guest Name + Country), and clicks
Check-in
Then the system generates a Registration Number (format: RR########), sets status to "In-
=== PAGE 31 ===
House," the room status updates to "Occupied" in Room Inventory, the guest billing account
is opened, and the record appears in the Search tab with a blue "In-House" badge.
AC-002 — Reservation-Linked Registration (Conditional Mandatory
Fields)
Given a pending reservation exists (e.g., GR00001252)
When the agent checks the Reservation checkbox, selects the reservation, and Meal Plan
and Reference are NOT selected
Then Check-in is blocked with validation errors on Meal Plan and Reference fields
When the agent selects both Meal Plan and Reference and completes all other required
fields
Then Check-in succeeds and the registration is linked to the reservation record.
AC-003 — Walk-in Registration: Meal Plan and Reference Optional
Given the agent is creating a walk-in registration (Reservation checkbox unchecked)
When the agent completes all truly required fields but leaves Meal Plan and Reference
unselected
Then Check-in succeeds without validation errors on those fields. No error is raised for
missing Meal Plan or Reference on walk-in registrations.
AC-004 — Listed Company Conditional Fields
Given the agent is on the Registration tab
When the Listed Company checkbox is unchecked
Then Contact Person, Mobile Number, Payment Mode, and Pay For fields are NOT visible in
the UI
When the Listed Company checkbox is checked and a company is selected
Then all four fields become visible and editable, and the company name appears in the
Listed Company dropdown.
AC-005 — Guest Count Enforcement
Given the Registration tab has 2 rooms with Adult count = 2 and Adult count = 1 (total adults
= 3)
When the agent adds only 2 Guest Detail records and clicks Check-in
Then Check-in is blocked with the message: "Guest records must match the declared adult
count. Please complete Guest Details."
When the agent adds the third Guest Detail record and clicks Check-in again
Then Check-in succeeds (assuming all other validations pass).
AC-006 — Rate Recalculation (RRC Modal)
Given the agent has selected a room with Rack Rate, Service Charge, VAT, City Charge, and
Additional Charges configured
When the agent clicks RRC, enters a target Total Room Rent Amount, selects all charge
component checkboxes, and clicks Apply Rate
Then the system back-calculates all individual charge components so that their sum equals
=== PAGE 32 ===
the entered Total Room Rent Amount, and all form fields are updated with the back-
calculated values.
AC-007 — Per-Room Date Override
Given the "Same as global date" checkbox is checked (default) and the header Check-In Date
is 2026-03-05
When the agent changes the header Departure Date to 2026-03-10
Then the per-room Departure Date auto-updates to 2026-03-10 and Number of Night
recalculates to 5
When the agent unchecks "Same as global date"
Then the per-room dates become independently editable and do not sync with further
header date changes.
AC-008 — Returning Guest Search and Pre-fill
Given a guest named "John Smith" with Date of Birth 01/01/1985 exists in the guest profile
store
When the agent enters "John Smith" in the Guest Details search and clicks Search
Then all form fields (Title, First Name, Last Name, Full Guest Name, DOB, Email, Phone,
Address, Passport, Visa, etc.) are pre-filled from the stored profile. The agent may modify
any pre-filled value before clicking Add.
AC-009 — Blocked Guest Warning
Given a guest profile has the Blocked Guest flag set to TRUE
When an agent retrieves this profile via the Guest Details Search function
Then a prominent warning alert is displayed to the agent before any further form action is
permitted. The agent must explicitly acknowledge the warning. The form fields are pre-filled
but the agent cannot proceed silently.
AC-010 — Complimentary Items: Package-Mandatory Cannot Be
Deselected
Given the selected Meal Plan defines "Wi-Fi Internet Access" as a mandatory complimentary
item
When the agent navigates to the Complimentary Item tab
Then "Wi-Fi Internet Access" checkbox is pre-checked and rendered as HTML disabled (non-
interactive). The agent cannot uncheck it.
When the agent clicks "Select All" to deselect all items
Then only non-mandatory items are deselected. "Wi-Fi Internet Access" remains checked.
AC-011 — Zero Complimentary Items (No Mandatory Package)
Given the selected Meal Plan has no mandatory complimentary items defined
When the agent navigates to the Complimentary Item tab and deselects all pre-checked
items (or none are pre-checked)
Then the registration proceeds to Check-in with zero complimentary items. No validation
error is raised for having no complimentary items selected.

AC-012 — Credit Card Data Masking and Tokenization
Given the agent is on the Others Information tab, Credit Card Information section
When the agent types a full 16-digit card number
Then the card number field masks the input and displays only the last 4 digits (e.g., ****
**** **** 1234) after the field loses focus
When the agent clicks Check-in and the registration is committed
Then no raw card number exists in any database table, audit log, or server log. Only the
tokenization service token and last 4 digits are stored.
AC-013 — Room Concurrency Conflict
Given Agent A and Agent B both open the Room List modal for the same Room Type and
select the same Room Number simultaneously
When Agent A completes Check-in first (commits the transaction)
Then Agent A's registration succeeds and the room status is set to "Occupied"
When Agent B subsequently attempts Check-in
Then Agent B receives the error: "Room [X] is no longer available. Please select a different
room." Agent B must return to the Room List modal to select a different room. Agent B's
session is not lost; all other form data is preserved.
AC-014 — Re-activate Registration (Manager Approval)
Given a registration has status "Checked Out" and the room is currently unoccupied
When a Front Desk Agent clicks Re-activate Registration
Then the system blocks the action with: "You do not have permission to re-activate
registrations."
When a Manager/Supervisor clicks Re-activate Registration, provides valid
credentials/approval code, and the room is available
Then the registration status is set back to "In-House," the room is marked "Occupied," and
an audit log entry is created capturing the approving manager's identity and timestamp.
AC-015 — Re-activate Registration (Room Conflict)
Given a Checked-Out registration exists for Room 1102 and Room 1102 is currently
"Occupied" by another In-House registration
When a Manager attempts to re-activate the Checked-Out registration
Then the system blocks re-activation with: "Room [1102] is currently occupied by another
guest. Please assign an alternative room before re-activating." The original registration
remains Checked Out.
AC-016 — Search Tab: Multi-Filter Search

Search
Then the results grid displays only records matching both criteria. The pagination footer
shows the correct count (e.g., "Showing 1 to 5 of 5 entries").
When the agent clicks Clear
Then all filters reset and the full unfiltered list is displayed.
